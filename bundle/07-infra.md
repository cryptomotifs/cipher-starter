# CIPHER Infra Playbook — 24/7 Solana Trading on Free-Tier

**Owner:** Head of Ops / SRE
**Scope:** Moves CIPHER from `cipher serve` on a Windows desktop to a 24/7 Oracle Cloud VM executing live Solana trades with $1 000 capital. Free-tier durable, zero monthly bill at steady state.
**Date:** 2026-04-17
**Stack baseline:** Python 3.11, FastAPI + uvicorn, SQLite/WAL, Next.js 15 frontend, existing `cipher serve` entrypoint, `docker-compose.yml`, GitHub Actions CI (ruff + mypy + pytest + tsc).

---

## 0. TL;DR

- **Host:** Oracle Cloud Always Free, `VM.Standard.A1.Flex` (4 OCPU ARM / 24 GB RAM / 200 GB block storage). Hetzner CX22 (≈ $4.59/mo) as 1-click contingency.
- **Runtime:** 3 systemd units (`cipher-bot`, `cipher-api`, `cipher-scheduler`) on a single VM. No Kubernetes. Docker Compose stays only for local dev; prod runs native Python to preserve Oracle ARM performance and systemd supervision.
- **Edge:** Cloudflare Tunnel fronts the API — no open public IP, TLS at edge.
- **Secrets:** Doppler CLI → env injected into each systemd unit via `EnvironmentFile=` generated by `doppler secrets download`.
- **Observability:** Sentry (errors) + Grafana Cloud OTLP (metrics/logs/traces) + BetterStack (uptime) + Healthchecks.io (cron heartbeats) + PostHog (product) + Resend (alert email). All free tier.
- **SLO:** **99 % monthly (≈ 7 h budget)**. 99.9 % would need paid redundant infra; not economic at $1 000 capital.
- **Monthly cost at $0 / $500 / $5 000 P&L:** **$0 / $0 / $0–$45** (first paid line is Neon or Sentry when volume crosses free caps).

## 1. Production Host Decision

### 1.1 Recommendation: Oracle Cloud Always Free (ARM)

**Shape:** `VM.Standard.A1.Flex`, 4 OCPU (Ampere Altra), 24 GB RAM, 200 GB boot+block, 10 TB egress/mo — all permanently free.

**Why it's right at this scale:**
- 4 ARM cores + 24 GB RAM absorbs: FastAPI, scheduler, trading loop, Kronos/CatBoost inference, Postgres-if-needed, without swapping.
- 10 TB egress covers any realistic signal volume plus Grafana/Sentry telemetry for years.
- Ubuntu 22.04 ARM has first-class Python 3.11 + aiosqlite 0.22.1 wheels; no source builds.
- Single region: we don't need multi-region for $1 000 capital.
- **Cost: $0/mo forever.**

### 1.2 Risks + mitigations

| Risk | Reality 2026 | Mitigation |
|---|---|---|
| Oracle reclaims idle "Always Free" VMs | Documented: must stay above ~10 % CPU or >20 % RAM averaged, or get billed/reclaimed after 7 d idle | Trading loop + scheduler + API keep CPU/RAM ≥ 15 % 24/7. Run `stress-ng --cpu 1 --cpu-load 20 --timeout 60s` nightly cron as insurance. |
| Account capacity errors on signup | Common in `us-ashburn-1`, `eu-frankfurt-1` | Pick a quiet region: `ca-toronto-1`, `me-jeddah-1`, `uk-london-1`. |
| Surprise account termination | Rare but real — reported on r/oracle | **Nightly off-VM backup to Cloudflare R2** (DB + `/etc` + systemd units + `git rev-parse HEAD`). Disaster-restore playbook tested monthly. |
| ARM-only wheels | Some legacy Python packages ship x86 only | Pinned `requirements.txt` already audited — all wheels (numpy, scipy, pandas, catboost, torch-cpu) have ARM64 manylinux wheels as of 2026-04. Confirm on a `manylinux_2_28_aarch64` base before pinning new deps. |
| Ingress SSH brute force | All public clouds face it | Cloudflare Tunnel (no public port 22). Oracle VCN security list: default deny inbound. Admin via `cloudflared access ssh`. |

### 1.3 Contingency: Hetzner CX22 (≈ $4.59/mo)

If Oracle reclaims: `hetzner-cx22.sh` one-shot bootstrap script (same Ansible role) rebuilds in 10 min. 2 vCPU / 4 GB / 40 GB — lighter but sufficient for our live footprint (peak ~1.8 GB). Shifts us to a small fixed bill but keeps us live.

## 2. Deployment Architecture

### 2.1 Process model — 3 systemd units

```
/etc/systemd/system/
├── cipher-bot.service         # trading loop — signal → guardrails → exec
├── cipher-api.service         # FastAPI (uvicorn) — dashboard, webhooks, Stripe
└── cipher-scheduler.service   # APScheduler — H6 nightly quality check, IC recompute
```

- **`cipher-bot.service`** — owns the trading loop, position state, circuit breaker, kill-switch file watcher (`/var/run/cipher/KILL`). `Restart=on-failure`, `RestartSec=10`, `StartLimitBurst=5`. If it flaps 5× in 5 min → unit enters `failed`, Sentry captures crash, P0 pager fires.
- **`cipher-api.service`** — `uvicorn cipher.api:app --host 127.0.0.1 --port 8080 --workers 2`. Cloudflare Tunnel binds to `127.0.0.1:8080`. `Restart=always`.
- **`cipher-scheduler.service`** — runs the existing `cipher.scheduler` with APScheduler; triggers H6 nightly 23:45 UTC; pings Healthchecks.io on each tick.

All three units:
- `User=cipher` (non-root, no shell)
- `EnvironmentFile=/etc/cipher/env` (generated by Doppler hook, chmod 0600)
- `WorkingDirectory=/opt/cipher/current` (symlink for blue-green — see §5)
- `ProtectSystem=strict`, `ProtectHome=yes`, `PrivateTmp=yes`, `NoNewPrivileges=yes`, `CapabilityBoundingSet=`
- Logging via `StandardOutput=journal` → `vector` sidecar → Grafana Cloud OTLP

### 2.2 Storage

- **Primary DB:** SQLite WAL at `/var/lib/cipher/cipher.db`. Already hardened by Sprint 14–16 (SyncDbEngine + AsyncDbEngine, bounded reader pool, `BEGIN IMMEDIATE`, nightly `wal_checkpoint(TRUNCATE)`).
- **Why SQLite holds at this scale:** single VM, single writer, <100 writes/sec in any realistic signal+trade mix. Avoids network-round-trip tax on hot path.
- **Migration trigger to Neon Postgres:** DB > 500 MB, or sustained > 200 writes/sec for 1 h, or we add a second app VM. Neon free = 0.5 GB / 190 compute-h per branch; upgrade $19/mo when crossed. Alembic migrations already in place (`migrations/`).
- **Trade journal:** separate append-only SQLite `trades.db` — never touched by hot path; dumped nightly to R2 as `trades-YYYY-MM-DD.db.gz` + CSV.

### 2.3 Networking

```
Internet
   │
   ▼
Cloudflare Edge ── TLS terminate ── WAF ── rate limit
   │
   ▼  (Cloudflare Tunnel, outbound from VM — no inbound ports)
cloudflared  ──►  127.0.0.1:8080 (uvicorn)
```

- Zero public ports open (VCN security list: all inbound deny).
- `cloudflared` runs as its own systemd unit, auto-renews token.
- Admin SSH via `cloudflared access ssh --hostname ssh.cipher.internal` (Cloudflare Access with email OTP + Google SSO).
- Egress: Solana RPC (Helius), Jupiter, Jito, Birdeye, Finnhub, etc. Cloudflare Gateway DNS policy blocks everything except allowlist.

### 2.4 Backups

- **DB:** `sqlite3 /var/lib/cipher/cipher.db ".backup /tmp/cipher-$(date +%F).db"` → `zstd -19` → `rclone copy` to Cloudflare R2 `s3://cipher-backups/db/`. Runs 03:00 UTC. 30-day rolling retention. Free R2 tier: 10 GB storage, 1 M Class A ops/mo — ample.
- **Config:** `/etc/cipher/` + `/etc/systemd/system/cipher-*.service` + `iptables-save` snapshot, same cron.
- **Git SHA + release tarball:** Copied to R2 after every successful deploy — enables 1-command rollback even if the VM dies.
- **Restore drill:** Monthly. `restore.sh <date>` on a throwaway Hetzner instance; must pass a replay test and be able to reach `/health` inside 15 min.

## 3. Monitoring + Alerting

### 3.1 Tooling map

| Tool | Purpose | Free tier used | Wiring |
|---|---|---|---|
| **Sentry** | Exceptions + perf traces | 5 k errors/mo, 10 k perf events | `sentry_sdk.init(dsn=SENTRY_DSN, traces_sample_rate=0.1)` in `cipher/bootstrap.py` — already hooked |
| **Grafana Cloud** | Metrics, logs, traces via OTLP | 10 k series, 50 GB logs, 50 GB traces, 14 d retention | `opentelemetry-sdk` → `OTLP_ENDPOINT`; log forwarder = `vector` reading journald |
| **BetterStack** | Black-box uptime + status page | 10 monitors / 3 min | HTTPS GET `https://api.cipher.dev/health`; assert `{"status":"ok"}` |
| **Healthchecks.io** | Cron heartbeats | 20 checks | `cipher-scheduler` pings after each nightly H6; `backup.sh` pings after R2 upload |
| **PostHog** | Product analytics | 1 M events/mo | Next.js frontend + server-side `capture("signal_viewed")` |
| **Resend** | Transactional email (P1/P2 digests + alert fallback) | 3 k emails/mo | `from=ops@cipher.dev` |
| **Telegram Bot** | P0 pager | Free | `telegram_bot.send(chat_id=OPS_CHAT, text=...)` via bot token |
| **Phone (Twilio)** | P0 fallback pager | Pay-as-you-go ≈ $0.02/SMS | Only when Telegram ACK not received in 3 min |

### 3.2 P0 — immediate pager (Telegram + SMS + email, parallel)

1. **Bot process crashed** — systemd `OnFailure=cipher-pager@%n.service` unit fires; Sentry also captures.
2. **Trading halted by circuit breaker** — `cipher.guardrails.CircuitBreaker.trip()` emits `alert.p0.breaker_tripped` metric + Sentry event with breach context.
3. **Wallet balance anomaly** — `cipher.wallet.watcher` diffs on-chain balance every 30 s; any change not tied to an `exec_id` in our journal → P0.
4. **Exchange / RPC degraded > 30 s** — Grafana alert `rate(rpc_error_total[1m]) > 0.5` for 30 s.
5. **Daily realized loss > threshold** (default: 5 % of capital = $50) — scheduler tick at end of each UTC hour; also live when a single trade closes > 3 %.
6. **Kill-switch engaged** — any process writing `/var/run/cipher/KILL` → confirmation pager.
7. **Clock drift > 2 s** — `chronyc tracking` drift triggers; Solana slot timing is unforgiving.
8. **SSL cert expiry < 14 d on api.cipher.dev** (Cloudflare-managed, defensive check).

Pager policy:
- Deliver concurrently to Telegram + email.
- If no ACK (`/ack <incident_id>` to bot) in 3 min → fire Twilio SMS to founder phone.
- If no ACK in 10 min → trigger auto-kill-switch (safer to halt than to keep running blind).

### 3.3 P1 — daily summary email 07:00 UTC

- Signals emitted (count, by asset class, by strength bucket).
- Trades executed + P&L (realized + unrealized).
- Error rate 24 h, deploy SHA currently running.
- Infra: CPU avg, RAM peak, disk free, WAL size, backup status.
- Top 3 slow endpoints from Grafana Cloud traces.

### 3.4 P2 — weekly retrospective email Mon 08:00 UTC

- Strategy attribution by layer (C1–C15 scorer P&L contribution).
- Best/worst 5 trades with links to signal detail.
- Uptime %, MTTR, incident list.
- SLO burn-rate chart snapshot.

## 4. Incident Response Runbook

Format designed for a half-asleep founder at 3 a.m. — every entry **one page, 6 sections**, pinned to fridge and in `/opt/cipher/RUNBOOK.md`.

### 4.1 Template

```
# P0 — <scenario name>
DETECT: <signal that fires — exact alert string>
PAGED VIA: <Telegram / SMS / email>
BLAST RADIUS: <money at risk in worst case>

# 30-SECOND TRIAGE (do this first, always)
1. ACK the pager: `/ack <incident_id>` in Telegram → silences re-alerts 15 min
2. Open dashboard: https://cipher.dev/ops (bookmarked on phone)
3. Check kill-switch state: curl https://api.cipher.dev/internal/kill-state

# DIAGNOSE (3 commands max)
$ ssh cipher-prod
$ systemctl status cipher-bot cipher-api cipher-scheduler
$ journalctl -u cipher-bot -n 200 --no-pager
$ tail -f /var/log/cipher/trades.log

# RECOVER
<scenario-specific: 3-5 steps max>

# CONFIRM FIXED
- /health returns 200 {"status":"ok"}
- Sentry: no new events for 5 min
- Grafana: circuit_breaker_state == 0
- Last heartbeat to Healthchecks < 90 s ago

# POST-INCIDENT (within 48 h)
- Fill template in docs/incidents/YYYY-MM-DD-<slug>.md
- Blameless 5-whys
- One action item, assigned + due date
```

### 4.2 Kill-switch — the universal 3 a.m. answer

```bash
ssh cipher-prod 'sudo touch /var/run/cipher/KILL'
# or from phone via Telegram bot:
/kill confirm
```

This halts **all new order submissions instantly** (hot-path file stat check, < 1 s latency). Existing positions keep their SL/TP orders on-chain. Resumes require human removal of the file AND a `/resume` confirm in Telegram (two-key).

### 4.3 Scenario-specific recovery summaries

| Scenario | Primary action | Backup action |
|---|---|---|
| Bot crashed | `systemctl restart cipher-bot`; if loop-crash → kill-switch + rollback `/opt/cipher/rollback.sh` | Redeploy from last R2 tarball |
| Circuit breaker tripped | **Do not override blind.** Read `cipher.guardrails` trip reason in journalctl; if correct → leave tripped, resumes after cooldown | If false-positive → set `CIPHER_BREAKER_FORCE_RESET=1`, restart bot, file bug |
| Wallet balance anomaly | Kill-switch immediately; rotate keys (see §6.4); check on-chain tx history for unknown sigs | Restore from Ledger cold-backup seed if hot key compromised |
| RPC degraded | Helius secondary RPC auto-fails over (configured in `.env: SOLANA_RPC_URL_FALLBACK`); if both down → kill-switch | Switch to QuickNode free-tier temp key, redeploy |
| Daily loss breach | Kill-switch fires automatically; manual review required before resume; scheduler refuses next-day start until `CIPHER_DAILY_LOSS_ACK=<YYYY-MM-DD>` set | n/a |
| Key compromise | Break-glass: §6.4 | n/a |

## 5. CI/CD Pipeline

### 5.1 Current state (Sprint 18)

`.github/workflows/ci.yml`: ruff + mypy + pytest + tsc on every PR to `main`. Green required to merge.

### 5.2 New production pipeline — 3 jobs, 1 file

```yaml
# .github/workflows/deploy.yml
jobs:
  test:        # existing ci.yml
  paper:       # on every push to main
    - deploy to staging VM (Hetzner CX11, $3.79/mo — one-time charge per month we ship)
    - run in paper mode against Solana devnet + live mainnet data (read-only signals)
    - smoke: /health + place 1 paper trade + close it + assert P&L
  prod:        # on release tag v*
    environment: production    # GitHub protection — manual approval gate
    - run Alembic migration dry-run on a local restore of prod DB snapshot
    - blue-green deploy via symlink flip (see §5.3)
    - post-deploy smoke: /health, scheduler tick, 1 signal round-trip
    - rollback if any smoke fails
```

### 5.3 Blue-green via symlinks (no Docker needed)

```
/opt/cipher/
├── releases/
│   ├── 2026-04-17-a1b2c3d/
│   ├── 2026-04-17-e4f5g6h/
│   └── 2026-04-18-i7j8k9l/  ← new
└── current -> releases/2026-04-18-i7j8k9l
```

Deploy script (`deploy.sh`):
1. Rsync new release into `releases/<date>-<sha>/`
2. `uv sync` inside release (UV handles locked wheels)
3. Run Alembic migration (atomic, guarded by `PRAGMA foreign_keys=OFF; BEGIN IMMEDIATE;`)
4. Smoke the new release on port 8081 (parallel uvicorn)
5. `ln -sfn releases/<new> current && systemctl reload cipher-*.service`
6. Wait 30 s, assert metrics; if fail → `ln -sfn releases/<previous> current` (rollback, ≤ 10 s)
7. Keep last 5 releases on disk.

### 5.4 DB migration gate

- Alembic migrations live in `migrations/`.
- CI job: `alembic upgrade head` against a fresh restore of latest prod snapshot from R2. Must pass.
- Prod deploy: migration runs **before** symlink flip. If migration fails → release aborted, no symlink change, no downtime.

## 6. Secrets + Keys in Production

### 6.1 Doppler wiring

One-time setup:
```bash
curl -Ls https://cli.doppler.com/install.sh | sh
doppler login
doppler setup --project cipher --config prd
```

Per-deploy generation (runs as systemd timer every 15 min — catches rotations without restart):

```ini
# /etc/systemd/system/cipher-secrets.service
[Service]
Type=oneshot
ExecStart=/usr/local/bin/doppler secrets download \
  --no-file --format env --project cipher --config prd \
  > /etc/cipher/env.new
ExecStartPost=/usr/bin/install -m 0600 -o cipher -g cipher /etc/cipher/env.new /etc/cipher/env
ExecStartPost=/bin/systemctl reload-or-restart cipher-bot.service cipher-api.service cipher-scheduler.service
```

```ini
# cipher-bot.service (excerpt)
EnvironmentFile=/etc/cipher/env
```

**Why not DOPPLER_TOKEN inside the service?** Because the service would need the token, and the chicken/egg forces us to either bake it into the unit file (bad) or pull at boot (race). The timer-based hydrate + `install -m 0600` pattern is the least-bad for single-VM.

### 6.2 Wallet key rotation

- **Hot trading wallet:** rotate **quarterly**, or immediately on any suspicion. Procedure: generate new keypair off-line, fund from cold, update Doppler, trigger secrets sync, transfer leftover SOL from old key, archive old key as `WALLET_HOT_RETIRED_<date>` in Doppler (audit trail).
- **Cold wallet:** hardware only (Ledger). Never touches the VM.
- **Principle:** max ever on hot wallet = 10 % of capital ($100 at this stage). Rest on Ledger.

### 6.3 Sentry DSN rotation if leaked

1. Sentry UI → Settings → Client Keys → Revoke → create new key.
2. Update Doppler `SENTRY_DSN`.
3. Secrets timer reloads within 15 min; or force: `systemctl start cipher-secrets.service`.
4. Verify: trigger a test exception; confirm it lands only in the new project.

### 6.4 Break-glass: key compromise at 3 a.m. Saturday

**Minimum viable procedure (≤ 15 min to safety):**

```bash
# 1. KILL: halt all trading instantly
ssh cipher-prod 'sudo touch /var/run/cipher/KILL'
# confirm from Telegram: /kill confirm

# 2. DRAIN: sweep hot wallet to cold
# requires cold-wallet signing on Ledger — procedure card in physical safe
python3 -m cipher.ops.emergency_sweep --dest <COLD_WALLET_ADDR>

# 3. REVOKE: new keypair + update Doppler
solana-keygen new -o /tmp/new-hot.json --no-bip39-passphrase
doppler secrets set WALLET_HOT_PRIVATE_KEY="$(cat /tmp/new-hot.json)" --config prd
doppler secrets set WALLET_HOT_RETIRED_$(date +%F)="<old key base58>" --config prd
shred -u /tmp/new-hot.json

# 4. ROTATE adjacent secrets (anything the compromised key could have pivoted to)
doppler secrets set HELIUS_API_KEY ... JITO_AUTH_KEY ... SENTRY_DSN ...

# 5. FORENSICS (do not skip, even at 3 a.m.)
#    Grab journalctl of last 24h to forensics bucket
ssh cipher-prod 'sudo journalctl --since "24h ago" | gzip > /tmp/forensics-$(date +%s).log.gz'
rclone copy cipher-prod:/tmp/forensics-*.log.gz r2:cipher-forensics/

# 6. RESUME only after a human review + postmortem draft started
```

The runbook card is laminated, lives next to the Ledger.

## 7. Logs Retention + Cost

### 7.1 Grafana Cloud free budget

- **Logs: 50 GB / 14 d retention** — at expected 2 GB/day (trades + signals + quote loops + heartbeats) we'd breach in ~3 weeks without sampling.
- **Metrics: 10 k active series** — generous; our ~1 500 series plan fits 6×.
- **Traces: 50 GB / 14 d** — use 10 % sample on hot path, 100 % on trade-execution spans.

### 7.2 Sampling rules (enforced in `structlog` processors)

| Event class | Retention policy | Rate |
|---|---|---|
| Trade execution, fills, P&L | **Always keep, full fidelity** | 100 % |
| Circuit breaker, kill-switch, wallet events | Always keep | 100 % |
| Errors / warnings | Always keep | 100 % |
| Signal generated/approved | Always keep | 100 % |
| Jupiter quote loops, RPC polls, price ticks | Sampled | 1 % (1-in-100) |
| Cache hits, heartbeats | Sampled | 0.1 % |
| Debug | Prod = dropped | 0 % (dev only) |

### 7.3 Mandatory structured-log fields

Every log line — enforced by `structlog` processor `require_fields` — must carry:

```
time        ISO-8601 UTC
level       debug | info | warn | error | critical
service     cipher-bot | cipher-api | cipher-scheduler
env         prod | staging | dev
release     <git sha>
event       snake_case event name
trace_id    W3C traceparent (for Grafana correlation)
asset       SOL / BTC / <symbol> (if applicable)
exec_id     if trade-related (correlates logs ↔ on-chain tx)
user_id     if user-scoped (respect PII scrub)
```

PII scrubber strips: wallet addresses (last 4 only), emails, Stripe IDs, API keys (any `sk_`, `pk_`, or 32+ char hex). Mandatory processor, unit-tested.

### 7.4 Overflow plan

If 50 GB/mo breached: first move is **further downsample quote loops to 0.1 %**, not upgrade. Upgrade to Pro ($8 / 100 GB extra) only after 3 consecutive breaches.

## 8. Uptime Target (SLO)

### 8.1 Recommendation: **99.0 % monthly** (≈ 7 h 18 min downtime budget)

**Reasoning:**
- $1 000 capital, < $50 daily loss cap — missing a signal is cheaper than paying for redundancy.
- Single-VM free-tier realistically delivers 99.0 – 99.5 %. Oracle maintenance windows, Cloudflare outages, and deploy-induced blips fit inside 7 h/mo.
- 99.9 % requires: multi-region, HA Postgres, paid RPC redundancy → $50+/mo baseline. Not economic until we're north of $10 k capital or external SLAs kick in.
- 95 % (36 h) is too lax — signals are time-sensitive; a multi-day outage bleeds signal relevance even without capital loss.

### 8.2 SLO definition (for Grafana alert rules)

```
SLI: ratio of ( /health 200-OK probes ∧ scheduler_heartbeat_fresh ∧ bot_loop_alive )
     over total probes in rolling 28-day window.

SLO: SLI ≥ 0.99

Burn-rate alerts:
  - Fast burn: 2% budget in 1h → P1 email
  - Medium:    5% budget in 6h → P1 email
  - Slow:     10% budget in 3d → P2 weekly entry
```

Error-budget policy: if budget < 25 % remaining mid-month → freeze non-critical deploys until month rolls over.

```
│       └── backup/               # rclone + R2 cron
├── scripts/
│   ├── deploy.sh                 # rsync + symlink + health check (called by CI)
│   ├── rollback.sh               # one-arg: previous | <release-id>

│   └── emergency_sweep.py        # break-glass hot→cold sweep
└── README.md                     # first-time setup (reads ≤ 5 min)
```

### 9.2 Docker vs. native — final call

Keep **`docker-compose.yml` for local dev only**. In prod, run native Python + systemd because:
- Systemd's `Restart=`, `OnFailure=`, cgroups (`MemoryMax=`), and journald integration outclass compose's restart policy.
- No container overhead on a 4-core VM we already own 100 % of.
- Deploy = rsync + symlink (10 s), not image build + registry push + pull (5+ min).
- Secrets wired once via `EnvironmentFile=` — no compose secret gymnastics.

Escape hatch: if we ever add a 2nd service VM, we switch to Nomad or Docker Compose + Watchtower, not K8s.
